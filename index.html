About Biippo

  • Car maintenance application developed for Android phones
  • Focuses on the 8th generation Honda Civic
  • Provides instructions for some of the most basic yet essential maintenance
  • Custom designed logo, buttons, and icons made using adobe illustrator

Biippo's User Focus

  • Chose 8th generation Honda Civic as first model due to the simplicity of maintenance and the popularity of the car model in order to reach a large user base
  • Designed a UI with user-friendliness and ease-of-use in mind
    • Appealing, welcoming color scheme
    • Custom and friendly icon design
  • Provided access to two different forms of instructions to allow users to choose their preferred style of learning
    • Step-by-step text instructions with images
    • Video instructions

Biippo's Technical Details

  • Application developed using Java in Android Studio
  • Created classes "Category" and "Subcategory" for the various maintenance categories
    • "Category" would contain an ArrayList of "Subcategory" instances
  • Pages would go through a switch statement of item IDs upon button selection to determine which intent, or activity, to create and start
  • Classes implemented "Parcelable", a fast form of serialization with more control designed specifically for Android applications
    • Watned to be able to store the "Category" and "Subcategory" objects on disk
  • Configured the various page layouts using XML files

About the Filesystem

  • Imitates a smaller subset of functions of a modern filesystem
  • Allows users to create a manipulate small-sized files
  • Provides a mechanism for storing these files in directories
    • Single root directory, not a recursive directory structure
  • Implemented on a simulated operating system

Filesystem's Technical Details

  • Filesystem implemented in Java using IntelliJ
  • Implemented a simplified version of the Unix Inode
    • Contains 12 pointers, 11 to indirect blocks and 1 to another set of blocks
    • Stores file length, number of file table entries, and a flag to determine the status of the Inode
    • Writes to disk, finds target block, registers index blocks, registers target blocks, and unregisters index blocks
  • Implemented a simplified directory
    • Contained a root directory
    • Abstracted class that finds free slots and allocates slots for files, and finds files for users
    • Achieves functionalities by interacting with Inodes
  • Implemented a file table
    • Maintains file entries in file cabinet through a vector of file table entries and a reference to the root directory
    • Operations include allocating and freeing files along with notifying users if the file table is empty or not
    • Handles the required operations for creating and configuring various Inodes
  • Implemented a Superblock
    • Describes important data of the whole filesystem and OS
    • Describes number of disc blocks, Inodes, and the block number of the head block of the free list
  • Implemented Kernel and SysLib
    • Kernel was used to enable the OS to handle the various operations required by the Filesystem
    • SysLib was used to send the proper Kernel interrupts for specific Filesystem functionalities

About the Jolly Banker

  • Mimics a banking system where various transactions may occur
    • Open account, withdraw funds, deposit funds, and print transaction history
  • Accounts for each customer has ten funds in which money can be placed
  • Certain accounts have overdraft protection that is covered by other predetermined accounts

Jolly Banker's Technical Details

  • Implemented in C++ using Visual Studio
  • Transactions would be read in a list from a given text file
    • Transactions would be stored in a queue and then processed
    • If statements were used to determine the transaction type
  • Accounts were stored in a Binary Search Tree
    • Accounts held lists of predetermined fund accounts
    • Accounts and funds to use for the individual transactions were determined by a code read in from the text file into the Transaction queue
  • The entire tree of accounts and their balances are printable
  • Individual account transaction histories were stored in queues and were printable

About the Disassembler

  • Decodes Opcodes and is displayed to the user
    • Only a subset of opcodes are decoded as they were the ones specified for the assignment
  • Handles opcodes that were not programatically decoded by printing "Bad Data" and continuing with the program logic
  • Provided a console as a user-friendly way to interact with the program
  • Could start within any valid address range and provide correct decoded output

Disassembler's Technical Details

  • Disassembled assembly code using the 68K Assembly instruction set
  • Implemented using the EASy68K IDE
  • Could decode a handful of the 68K Assembly instruction set
    • Decoded instruction set seen below:
  • Uses a process of elimination through bit shifting to determine the current opcode that is being decoded
    • Begins by looking at the first 4 bits of specific instructions
    • Uses a jump table to jump to the next set of logic based on the given bits, or to print "Bad Data" if the bits are invalid
  • Created our own very extensive test to verify that all possible combinations of required opcodes could be decoded properly
  • Created and used a java application that could parse some of the output from decoding in order to compare the results against the original instructions for verification
  • Console application for users was developed using the 68K instruction set as well
